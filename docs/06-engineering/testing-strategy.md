# 测试框架与质量策略

## 1. 目标

本文档定义 Bifrost 第一阶段的测试框架、测试分层、验收方式和质量门禁。

目标是保证：

- 权限策略不会误放行
- 设备信任逻辑可靠
- 网关代理行为可验证
- 客户端与后台 UI 可稳定回归
- 本地多容器环境能模拟真实私有服务访问

## 2. 测试分层

第一阶段建议采用以下测试层级：

```text
Unit Tests
Integration Tests
Contract Tests
E2E Tests
Manual Security Checks
```

每层职责不同，不应互相替代。

## 3. 服务端测试

## 3.1 单元测试

使用：

- Go `testing`
- Go 标准库断言或轻量辅助工具

必须覆盖：

- 策略判断
- 用户级允许/禁止覆盖
- 角色授权
- 设备状态判断
- 服务状态判断
- 错误码映射

关键用例：

- 用户禁用时拒绝
- 设备禁用时拒绝
- 服务禁用时拒绝
- 用户级 deny 优先级高于 allow
- 用户级 allow 优先级高于角色默认拒绝
- 无任何授权时默认拒绝

## 3.2 集成测试

使用：

- Go `testing`
- 本地 PostgreSQL 容器
- 测试专用 migration

必须覆盖：

- 用户创建与查询
- 角色服务授权
- 用户级覆盖写入与读取
- 设备注册与查询
- 会话创建与吊销
- 审计写入与查询

## 3.3 网关代理测试

使用：

- Go `httptest`
- Docker Compose 中的 mock upstream 服务

必须覆盖：

- 授权用户访问成功
- 未授权用户访问被拒绝
- 服务禁用后访问被拒绝
- 上游不可达返回 `502`
- 上游超时返回 `504`
- 路径改写正确
- query string 保留
- 访问审计写入

## 4. 前端测试

## 4.1 单元测试

使用：

- Vitest
- React Testing Library，若引入

覆盖对象：

- API client 错误处理
- 权限状态展示
- 表单校验
- 纯函数工具

## 4.2 组件测试

重点组件：

- Button
- Input
- Table
- Drawer
- Dialog
- Status Badge
- Error State
- Empty State

检查点：

- Light / Dark 样式可读
- 禁用态明确
- loading 态不跳动
- 错误状态展示 requestId

## 4.3 E2E 测试

使用：

- Playwright

后台 E2E 场景：

- 管理员登录
- 创建用户
- 创建角色
- 创建服务
- 设置角色授权
- 设置用户级 deny
- 查询审计日志

客户端 E2E 场景：

- 用户登录
- 设备注册
- 查看服务列表
- 打开服务入口
- 设备禁用后提示不可用

Electron E2E 第一阶段可以先覆盖关键流程，不追求全平台全自动。

## 5. 契约测试

API 契约以 OpenAPI 为源。

必须验证：

- 服务端响应符合统一响应结构
- 错误码属于已定义枚举
- 分页结构统一
- 前端生成类型与 OpenAPI 一致

禁止：

- 接口返回裸数组
- 某些接口返回 `rows`，另一些返回 `items`
- 未登记错误码直接返回给前端

## 6. 多容器联调测试

本地开发环境至少需要模拟：

- PostgreSQL
- Gateway Server
- Admin Web
- Mock GitLab
- Mock Jenkins
- Mock Docs

联调目标：

- 后台配置多个服务
- 客户端获取授权服务列表
- 网关按服务 key 转发到不同容器
- 未授权服务拒绝
- 审计日志可查

## 7. 安全回归用例

必须长期保留：

- 客户端传入任意 upstream URL 时被拒绝
- 未配置服务 key 被拒绝
- 禁用设备不能访问
- 禁用用户不能访问
- 服务禁用后所有用户不能访问
- 用户级 deny 覆盖角色 allow
- 上游不可达不应误报权限不足

## 8. 质量门禁

进入合并前至少要求：

- Go 单元测试通过
- Go 集成测试通过
- 前端单元测试通过
- API 契约检查通过
- 关键 E2E 场景通过

文档阶段先定义门禁，具体命令在实现阶段落到 `package.json`、`turbo.json` 和 Makefile 或 Taskfile 中。

## 9. 测试数据策略

建议提供固定测试数据：

- 管理员账号
- 普通用户账号
- 研发角色
- 运维角色
- GitLab 服务
- Jenkins 服务
- Docs 服务
- 一台可信设备
- 一台禁用设备

测试数据应可重复初始化，不能依赖手工点击创建。

## 10. 验收结论标准

第一阶段开发完成时，测试应证明：

- 授权用户可以访问授权服务
- 未授权用户不能访问未授权服务
- 设备禁用立即生效
- 服务禁用立即生效
- 多个 Docker 上游服务可以被网关正确区分和转发
- 所有关键行为有审计记录
