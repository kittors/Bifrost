# 开发规范与协作原则

## 1. 总原则

Bifrost 项目开发遵循以下原则：

- 安全优先
- 契约优先
- 设计系统优先
- Monorepo 优先
- 可维护优先
- 最小侵入优先

这些原则高于单次“写起来更快”的局部便利。

## 2. 阶段约束

在设计阶段：

- 先统一文档、架构、规范、设计系统和契约
- 未进入实现阶段前，不应提前生成业务脚手架

在实现阶段：

- 先建共享能力，再建应用业务
- 先建契约，再建接口消费
- 先建 token，再建页面样式

## 3. Monorepo 协作原则

- 客户端与后台必须共享同一套设计 token
- 客户端与后台必须共享同一套基础组件体系
- 契约、错误码、分页结构必须统一放入共享包
- 不允许应用各自复制同一份工具函数或常量定义

## 4. 前端开发规范

### 4.1 技术边界

- 统一使用 `React + TypeScript + Tailwind CSS v4 + shadcn/ui`
- 统一使用 `pnpm workspace + Turborepo`
- 统一使用 `Biome`

### 4.2 样式规范

- 默认使用 Tailwind utilities
- 尽量不用原生 CSS
- 不使用 SCSS、Less、CSS Modules、styled-components
- 页面中重复样式应抽象为共享组件或受控变体

### 4.3 组件规范

- 基础交互组件沉淀到 `packages/ui`
- 业务页面不直接堆积长 class 串复制粘贴
- 所有基础组件必须支持 Light / Dark
- 所有组件必须遵守统一的尺寸、字号、圆角、边框规范

## 5. 服务端开发规范

### 5.1 标准库优先

第一阶段优先采用 Go 标准库解决：

- HTTP 服务
- TLS
- 反向代理
- 并发模型

只在收益明确时引入轻量第三方依赖。

### 5.2 目录与边界

- 以业务能力划分 `internal` 目录
- 不做超大统一 handler 文件
- 授权判断与反向代理必须分离
- 审计记录必须通过统一入口

### 5.3 错误处理

- 错误码统一管理
- 错误对外结构统一
- 服务端日志与 API 错误响应保持可关联

## 6. API 与契约规范

- OpenAPI 作为统一契约源
- 共享契约包中维护错误码、分页结构与公共 schema
- 前端禁止随意手写与后端不同的 DTO
- 新增接口前先定义契约，再实现逻辑

## 7. 安全开发规范

- 不在代码中硬编码敏感密钥
- 不将私钥明文落盘
- 不暴露任意目标地址代理能力
- 所有高风险操作都必须审计
- 所有鉴权与授权逻辑都应可追溯

## 8. 文档规范

以下变更必须同步文档：

- 设计系统变化
- API 契约变化
- 错误码变化
- 授权模型变化
- 依赖版本基线变化
- 部署方式变化

文档不是补充材料，而是工程事实的一部分。

## 9. 测试规范

### 前端

- 组件与逻辑使用 `Vitest`
- 关键页面流转使用 `Playwright`

### 服务端

- 核心授权逻辑必须有单元测试
- 反向代理关键行为必须有集成测试
- 会话、设备、策略逻辑必须覆盖异常路径

## 10. Git 与提交规范

建议：

- 小步提交
- 提交消息语义清晰
- 一次变更只解决一个主要问题

推荐前缀：

- `docs:`
- `feat:`
- `fix:`
- `refactor:`
- `chore:`
- `test:`

## 11. 依赖管理规范

- 所有依赖版本以基线文档为准
- 不允许随意升级核心框架
- 升级前必须评估对设计系统、Electron、契约生成、构建链的影响
- 同类能力避免引入多个重叠库

## 12. 设计评审原则

新增页面或组件必须评审以下问题：

- 是否符合清爽、紧凑、低干扰方向
- 是否使用了统一 token
- 是否在 Light / Dark 下都可读
- 是否适合三端并行桌面场景与后台场景
- 是否增加了不必要的视觉复杂度

## 13. 架构红线

- 不得把第一阶段问题扩展为全局 VPN 问题
- 不得为了省事接管系统代理或路由
- 不得绕过统一契约自由定义响应结构
- 不得让客户端和后台分别长出两套设计系统
