# Electron 客户端 UI 规范

## 1. 设计定位

客户端是“小卡片式私有服务访问入口”，不是完整管理后台，也不是大工作台。它应当清爽、紧凑、稳定，让用户快速完成：

- 登录
- 查看可访问服务
- 打开服务
- 了解设备与会话状态

## 2. 窗口规范

默认窗口：

- 宽度：`420px`
- 高度：`560px`
- 最小宽度：`380px`
- 最小高度：`480px`

窗口策略：

- 使用原生窗口边框
- 内容区自定义
- 不做复杂自绘标题栏
- 不隐藏系统窗口控制按钮

原因：

- 三端并行更稳定
- 减少窗口拖拽、最大化、全屏问题
- 降低平台差异成本

## 3. 主布局

建议布局采用单卡片纵向结构：

```text
┌──────────────────────┐
│ Header               │
├──────────────────────┤
│ Connection Status    │
├──────────────────────┤
│ Service List         │
├──────────────────────┤
│ Account / Settings   │
└──────────────────────┘
```

尺寸：

- Header：`44px`
- 页面 padding：`16px`
- 区块间距：`12px`
- 服务列表行高：`40px`

不使用后台式左侧导航，不使用大面积内容工作区。

## 4. Header

内容：

- 当前连接状态
- 当前用户
- 设备状态
- 设置入口或更多菜单

状态展示：

- 已连接
- 未连接
- 会话即将过期
- 设备已禁用

Header 不应放过多操作按钮。

## 5. 导航策略

客户端不使用常驻 Sidebar。导航应尽量轻量：

- 默认首页就是服务列表
- 账号、设置、诊断放在底部或更多菜单中
- 如需分段，可使用紧凑 Tabs
- Tabs 高度 `32px`

目标是让客户端像一个轻量控制卡片，而不是后台管理系统。

## 6. 服务列表

展示方式：

- 默认使用紧凑单列列表
- 支持搜索和分组

服务卡片内容：

- 服务图标
- 服务名
- 分组
- 状态
- 权限来源
- 最近访问

卡片样式：

- 服务项圆角 `10px`
- 服务项边框 `1px`
- 默认无重阴影
- hover 时轻背景变化
- 单个服务项高度建议 `40px` 到 `44px`

## 7. 登录页

布局：

- 单卡片紧凑面板
- 面板宽度随窗口，内容宽度约 `360px` 到 `388px`
- 输入框高度 `40px`
- 登录按钮高度 `40px`

字段：

- 服务端地址
- 用户名
- 密码

文案：

- 简短明确
- 错误提示保留 requestId

## 8. 设备状态展示

设备状态必须清晰可见：

- 未注册
- 已信任
- 已禁用
- 验证失败

设备详情建议展示：

- 设备名称
- 操作系统
- 客户端版本
- 公钥指纹后 8 到 12 位
- 最近登录时间

## 9. 打开服务交互

用户点击服务时：

1. 客户端请求服务访问入口
2. 若允许，打开系统默认浏览器
3. 若拒绝，展示拒绝原因

按钮文案建议：

- “打开服务”
- “重新登录”
- “查看设备状态”

## 10. 错误与空状态

空状态：

- 无可访问服务
- 无网络连接
- 设备未绑定

错误状态：

- 登录失效
- 设备被禁用
- 服务不可用
- 权限不足

错误组件必须展示：

- 用户可读说明
- 建议操作
- requestId，若存在

## 11. 主题

客户端必须支持：

- Light
- Dark
- 跟随系统，后续可实现

第一阶段至少实现 Light / Dark 手动切换。

## 12. 交互密度

建议：

- 列表行高 `40px`
- 小按钮 `32px`
- 默认按钮 `36px`
- 登录按钮 `40px`

避免：

- 大面积 Hero
- 高度超过 `48px` 的常规按钮
- 高饱和大色块
- 后台式侧边栏
- 复杂多列布局

## 13. 三端适配注意事项

macOS：

- 避免把关键按钮放到窗口左上角附近

Windows：

- 保持焦点态清晰
- 不依赖 macOS 风格毛玻璃

Linux：

- 字体 fallback 必须可靠
- 避免平台特定托盘行为作为唯一入口

## 14. 验收标准

- 三端窗口行为正常
- 登录页紧凑清晰
- 服务列表扫描效率高
- 设备禁用等安全状态醒目
- 不产生平台视觉割裂
- 客户端整体观感像一个轻量小卡片，而不是后台控制台
