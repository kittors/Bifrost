# 设计原则与界面风格

## 1. 设计目标

Bifrost 的界面应体现以下关键词：

- 清爽
- 紧凑
- 稳定
- 克制
- 高效

这是一套偏“工作台 / 控制台”的系统，不是营销站，也不是强调情绪化品牌表达的产品。设计需要优先服务于：

- 状态判断
- 快速扫描
- 配置效率
- 错误识别
- 风险感知

## 2. 整体视觉方向

推荐视觉语言：

- 浅背景 + 细边框 + 低阴影
- 中性灰为基础，蓝青色作为品牌语义主轴
- 信息密度较高，但不拥挤
- 强对齐，少装饰
- 圆角适中，不做夸张大圆角

明确不采用以下风格：

- 大面积渐变营销风
- 厚重拟物风
- 高饱和霓虹科技风
- 紫色主导视觉
- 卡片堆叠式仪表盘

## 3. 客户端界面原则

桌面客户端更接近“服务入口工作台”，重点不在长流程编辑，而在于：

- 登录
- 状态确认
- 查看可访问服务
- 快速打开服务
- 查看设备与会话信息

因此客户端应：

- 使用稳定的双栏或三段布局
- 避免复杂多层导航
- 避免大面积留白
- 避免沉重卡片感
- 优先使用列表、状态条、工具栏、紧凑详情区

## 4. 后台界面原则

Web 管理后台更接近“安全控制台”，重点在：

- 策略配置
- 对象管理
- 设备封禁
- 审计回溯

因此后台应：

- 优先表格、筛选器、抽屉、时间线
- 页面结构以“筛选 -> 数据 -> 操作 -> 详情”为主
- 首页不是营销 dashboard，而是工作 dashboard
- 强化风险状态与审计反馈

## 5. 三端并行适配原则

项目需要同时考虑 `macOS`、`Windows`、`Linux`，因此设计上应避免平台偏置。

要求：

- 不依赖某个系统专有视觉语言作为核心卖点
- 组件尺寸、图标、字体在三端都要稳定
- 不设计只适配某单一系统标题栏的极端方案
- Electron 客户端优先使用原生窗口边框 + 自定义内容区

## 6. 信息密度原则

本项目需要“紧凑”，但不能压缩到难读。

建议：

- 表格默认密集但不极端
- 行高控制在高效扫描区间
- 保留清晰分组和留白节奏
- 一页优先展示有效信息，而不是大面积空白装饰

不建议：

- 超宽按钮
- 超高卡片
- 大面积 Hero 区块
- 为了“高级感”牺牲配置效率

## 7. 交互状态原则

状态必须清晰、一致、可预期。

关键状态包括：

- 默认
- hover
- active
- focus-visible
- disabled
- loading
- success
- warning
- danger

要求：

- 焦点态必须明显
- disabled 态必须一眼可识别
- 危险操作必须和普通主按钮明显区分
- loading 不应引发布局跳动

## 8. 组件设计原则

### 统一性

- 按钮高度统一
- 输入框高度统一
- 表格风格统一
- Drawer / Modal / Popover 统一

### 可组合性

- 基础组件负责样式与常见状态
- 业务组件负责组合与语义
- 不让业务页面自己拼接零散样式

### 可审计感

对于安全产品，UI 应当帮助用户理解“当前状态是否可信”。因此：

- 设备状态、账号状态、服务状态必须可见
- 权限不足要有明确拒绝信息
- 风险操作必须有确认与结果反馈

## 9. 视觉层级原则

层级依靠以下因素建立，而不是靠夸张效果：

- 字号
- 字重
- 背景层
- 边框
- 间距
- 语义色

默认不依赖：

- 重投影
- 大渐变
- 粗边框
- 大色块背景

## 10. 主题原则

Light 与 Dark 都必须是一等公民。

要求：

- 不是简单颜色反相
- 两套主题都要保持“清爽、紧凑、低干扰”
- Dark 主题不能纯黑
- Light 主题不能纯白
- 两套主题都应保证文本和状态色可读性

## 11. 内容表达原则

文案风格建议：

- 短句
- 明确
- 偏操作语言
- 减少模糊措辞

例如：

- “设备已被禁用”
- “你没有访问该服务的权限”
- “登录状态已失效，请重新登录”

避免：

- 大段解释性文案
- 情绪化提示
- 含糊不清的错误描述

## 12. 设计红线

- 不使用随机圆角和随机高度
- 不使用多个不一致的蓝色主色
- 不在业务组件里自由定义颜色与阴影
- 不使用大面积原生 CSS 重写 Tailwind 体系
- 不让客户端和后台形成两套不同视觉语言
