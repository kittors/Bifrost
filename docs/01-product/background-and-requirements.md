# 项目背景与需求说明

## 1. 项目背景

公司目前在云服务器上部署了多种私有服务，这些服务对日常研发、交付、协作和内部运维至关重要，例如：

- 私有化部署的 `GitLab`
- `Jenkins`、制品库、文档系统
- 内部管理后台
- 部署在宿主机端口或 Docker 容器内部的 Web 服务

这些服务具有明显的内部属性，不适合直接暴露在公网中供任意来源访问。公司希望做到：

- 私有服务不对公网匿名开放
- 外网电脑只有在安装了公司指定客户端并成功登录后，才能访问被授权的私有服务
- 不同账号、角色、设备拥有不同访问范围
- 所有关键访问都可审计、可追溯、可封禁

因此，本项目需要构建一套用于“受控访问私有 Web 服务”的统一平台。

## 2. 为什么要做这个项目

## 2.1 当前问题

如果直接把私有服务暴露到公网，会带来以下问题：

- 任意来源都可以探测和尝试访问私有服务
- 即使服务本身需要登录，也会暴露服务指纹、登录入口、弱口令攻击面
- 内部服务配置通常并不是为公网高强度对抗设计的
- 无法从统一入口控制“谁可以访问哪些服务”
- 无法统一记录访问行为、设备状态、配置变更
- 不同服务各自维护权限，运维成本高且容易失控

## 2.2 业务需求本质

本项目不是单纯做一个代理工具，也不是做一个通用 VPN，而是做一套企业内部服务访问控制平台。它解决的是以下问题：

- 如何让外网员工安全访问指定私有服务
- 如何在“最小暴露面”的前提下开放访问能力
- 如何确保授权、设备、审计、吊销都由统一平台控制
- 如何避免客户端破坏用户本机已有网络环境

## 2.3 为什么不做全局 VPN

本项目第一阶段明确不做全局 VPN、Overlay 网络或透明代理，原因包括：

- 用户本机可能已经存在其他 VPN、隧道或代理软件
- 全局路由、系统 DNS、TUN/TAP、系统代理接管会显著增加冲突与排障成本
- 业务目标只是访问被授权的私有 Web 服务，而不是获得整张内网能力
- 企业安全上更需要“按服务授权”，而不是“接入即全网访问”

因此，本项目第一阶段采用“统一访问网关 + 显式服务授权”的方案。

## 3. 项目目标

## 3.1 核心目标

- 为企业私有 Web 服务建立统一受控访问入口
- 要求外网设备必须安装客户端并登录后才能访问服务
- 通过账号、角色、设备和服务策略共同决定访问权限
- 所有关键事件必须具备审计能力
- 在不干扰用户本机 VPN/代理的前提下完成访问控制

## 3.2 用户体验目标

- 客户端登录流程清晰、简洁、安全
- 用户可以在客户端中看到自己可访问的服务列表
- 用户点击服务后即可在浏览器中打开受控入口
- 客户端界面清爽、紧凑、低干扰
- Web 管理后台适合高频运营与配置操作

## 3.3 工程目标

- 采用统一 Monorepo 管理客户端、后台、共享包与服务端文档
- 统一设计系统、统一 API 响应结构、统一错误码
- 采用现代稳定技术栈，且精确锁定版本基线
- 第一阶段架构清晰，便于后续扩展 SSH/TCP/SSO 等能力

## 4. 第一阶段范围

第一阶段范围已经确认如下：

- 支持访问 `HTTP/HTTPS` 私有服务
- 支持宿主机端口形式的服务
- 支持 Docker 容器中对内暴露、由网关转发的服务
- 支持后台配置用户、角色、设备、服务、策略
- 支持角色授权服务
- 支持用户级允许/禁止覆盖
- 支持设备注册、设备绑定、设备禁用
- 支持客户端登录与会话维持
- 支持统一访问网关与反向代理
- 支持审计登录、访问、配置变更
- 支持 Light / Dark 主题

## 5. 非目标

以下能力不属于第一阶段交付目标：

- 全局 VPN
- Overlay 网络
- 系统级透明代理
- 系统级 DNS 接管
- TUN/TAP 虚拟网卡
- SSH、数据库、原始 TCP 代理
- OIDC、SSO、LDAP、AD 集成
- 浏览器插件
- 移动端客户端
- 审批流、工单流、多租户 SaaS 化能力

## 6. 关键角色

## 6.1 普通用户

- 安装桌面客户端
- 使用后台配置的账号密码登录
- 查看自己有权限访问的服务
- 通过客户端打开服务入口

## 6.2 管理员

- 创建与维护用户账号
- 维护角色与角色权限
- 配置服务目录与内网目标地址
- 查看设备状态
- 禁用设备、禁用用户、封禁服务
- 查看审计日志

## 6.3 平台维护者

- 部署和维护网关服务端
- 配置证书、数据库、网关规则
- 维护版本与发布策略
- 处理运行告警与安全事件

## 7. 关键约束

## 7.1 网络约束

客户端必须避免干扰用户本机已有网络环境。明确约束为：

- 不修改系统代理
- 不修改系统 DNS
- 不修改系统路由
- 不创建 VPN/TUN/TAP
- 不要求用户禁用现有 VPN 或代理软件

## 7.2 安全约束

- 所有传输必须走加密通道
- 设备身份必须可验证、可吊销
- 访问授权必须经过统一判断
- 服务访问必须可审计
- 管理员关键操作必须可追溯

## 7.3 产品约束

- 第一阶段只做 Web 服务访问，不把问题泛化成“网络接入平台”
- UI 风格必须紧凑、清爽、低干扰
- 尽量少写原生 CSS，完全采用 Tailwind CSS v4 的 utility-first 方式
- 客户端与后台必须基于同一个 Monorepo 与同一套设计系统

## 8. 成功标准

若第一阶段上线后达到以下结果，可视为成功：

- 外网用户只有在安装客户端并登录后，才能访问被授权服务
- 未授权用户、未绑定设备、被禁用设备均无法访问对应服务
- 管理员可以通过后台独立完成用户、角色、设备、服务配置
- 用户可从客户端顺畅打开 `GitLab` 等私有服务
- 客户端不影响本机其他 VPN 或代理软件正常工作
- 登录、访问、配置变更均可在审计日志中查询
- 客户端和后台界面风格统一、信息密度合理、交互清晰

## 9. 推荐实施策略

第一阶段建议围绕以下顺序推进：

1. 先定义统一设计系统、API 契约与 Monorepo 结构
2. 再实现后台的用户、角色、服务、设备管理能力
3. 同步实现网关的认证、策略判断和反向代理
4. 最后实现桌面客户端登录、设备注册、服务入口打开流程

这样可以保证平台的管理能力、访问控制能力和客户端体验从一开始就是统一闭环，而不是后期拼装。
