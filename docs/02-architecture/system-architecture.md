# 系统架构设计

## 1. 总体架构

Bifrost 第一阶段采用“三端一通道”架构：

- `desktop-client`：桌面客户端
- `gateway-server`：统一访问网关与服务端核心
- `admin-web`：Web 管理后台
- `secure-access-channel`：由客户端登录状态、设备身份与网关会话共同构成的受控访问通道

该架构的核心不是为用户建立整张内网的访问能力，而是为每个被授权的私有 Web 服务建立一个显式、可控制、可审计的访问入口。

## 2. 核心组件职责

## 2.1 desktop-client

客户端职责：

- 用户登录
- 设备首次注册与设备绑定
- 安全保存本机设备身份与刷新令牌
- 展示当前用户可访问的服务列表
- 打开受控服务入口
- 展示登录状态、设备状态、会话状态

客户端边界：

- 不做系统级代理
- 不做系统级 DNS 接管
- 不做路由劫持
- 不做透明转发
- 不直接承担反向代理职责

## 2.2 gateway-server

服务端职责：

- 账号认证
- 设备验证
- 会话管理与令牌签发
- 服务目录管理
- 权限策略判断
- 反向代理转发
- 审计事件记录
- 后台 API 提供

服务端是系统安全核心。用户对私有服务的访问必须经过它的认证与授权判断。

## 2.3 admin-web

后台职责：

- 用户管理
- 角色管理
- 用户级允许/禁止覆盖
- 设备管理
- 服务目录管理
- 服务目标地址配置
- 审计查看
- 风险封禁与停用操作

后台本身不转发业务流量，只调用服务端管理 API。

## 3. 访问模型

## 3.1 访问原则

- 所有服务访问都必须显式命中“已配置服务”
- 所有访问都必须基于已登录会话
- 所有访问都必须经过设备状态校验
- 所有访问都必须经过角色与用户覆盖规则判断
- 默认拒绝，显式允许

## 3.2 典型访问流程

1. 用户打开桌面客户端
2. 用户使用后台分配的账号登录
3. 客户端生成或读取设备密钥对
4. 服务端验证账号与设备身份，建立会话
5. 客户端获取“当前用户可访问服务列表”
6. 用户点击某个服务，例如 `GitLab`
7. 客户端打开受控入口，例如 `https://gateway.company.com/s/gitlab/...`
8. 网关校验令牌、设备、角色、用户覆盖规则、服务状态
9. 校验通过后，网关将请求转发到目标内网服务
10. 网关记录访问审计日志

## 4. 服务目录模型

第一阶段中，服务目录是整个系统的关键抽象。每个服务至少具备以下属性：

- 服务 ID
- 服务显示名称
- 协议类型，第一阶段仅 `http` 或 `https`
- 服务分组
- 内网目标地址
- 是否启用
- 访问说明
- 图标或标识
- 可见性与授权范围

推荐的抽象方式：

- 一个“服务”代表一个被授权访问的业务入口
- 一个“服务”对应一个固定目标上游
- 网关按服务 ID 解析目标地址，不允许客户端直接传入任意内网 URL

这样可以避免把网关做成任意目的代理。

## 5. 权限模型

采用“角色为主，用户覆盖为辅”的混合模型。

决策顺序建议如下：

1. 用户和设备必须处于启用状态
2. 服务必须处于启用状态
3. 用户级禁止优先级最高
4. 用户级允许高于角色默认结果
5. 若无用户覆盖，则按角色授权判定
6. 若仍无命中，则默认拒绝

这种模型适合：

- 用角色批量维护常规权限
- 用用户级覆盖处理临时放行、临时封禁、例外账号

## 6. 部署形态

## 6.1 外部入口

公网只暴露少量受控入口：

- 网关域名
- 管理后台域名

不应直接暴露私有服务自身域名或端口。

## 6.2 内部连接

网关通过内网连接到目标服务，例如：

- 访问宿主机端口
- 访问容器编排内部服务
- 访问内网域名

第一阶段推荐将“服务发现与上游配置”直接内置在 Go 服务端中，不额外拆分独立服务。

## 7. 可扩展性边界

为了避免过度设计，第一阶段不实现以下扩展，但架构需要预留边界：

- OIDC / LDAP / SSO 身份集成
- SSH / TCP 代理
- 审批流与临时授权
- 多因素认证
- 高级风险引擎

需要预留的扩展点包括：

- 身份源抽象层
- 协议适配层
- 策略引擎扩展点
- 审计事件结构化模型

## 8. 关键架构决策

## 8.1 不实现全局 VPN

原因：

- 容易与用户本机现有 VPN / 代理冲突
- 超出第一阶段业务目标
- 引入过大的网络行为与安全复杂度

## 8.2 标准库优先

服务端第一阶段优先基于 Go 标准库构建 HTTP 服务、TLS 与反向代理能力，只为明确收益场景引入轻量第三方依赖。

## 8.3 契约优先

客户端、后台、服务端共享一份 API 契约与错误码定义，避免多端漂移。

## 8.4 Monorepo 优先

由于设计系统、共享类型和 API 契约都需要统一维护，本项目从一开始就采用 Monorepo。
