# 数据流与访问链路

## 1. 目标

本文档描述 Bifrost 第一阶段核心数据流，包括登录、设备注册、服务列表、服务访问、后台配置变更和审计记录。

## 2. 总体链路

核心链路为：

```text
Desktop Client -> Gateway Server -> Private Web Service
Admin Web -> Gateway Server -> Database
Gateway Server -> Audit Log -> Database
```

私有 Web 服务不直接暴露给公网用户，所有访问通过 Gateway Server。

## 3. 登录数据流

1. 用户在客户端输入账号密码
2. 客户端向服务端提交登录请求
3. 服务端校验账号密码
4. 服务端检查用户状态
5. 服务端发起设备挑战
6. 客户端使用设备私钥签名挑战
7. 服务端验证设备签名和设备状态
8. 服务端签发访问令牌和刷新令牌
9. 服务端记录登录审计事件

失败分支：

- 账号密码错误：`AUTH_INVALID_CREDENTIALS`
- 用户禁用：`USER_DISABLED`
- 设备禁用：`DEVICE_DISABLED`
- 设备签名失败：`DEVICE_KEY_INVALID`

## 4. 首次设备注册数据流

1. 客户端发现本机无设备密钥
2. 客户端生成设备密钥对
3. 用户登录账号
4. 客户端提交设备公钥和设备信息
5. 服务端创建设备记录
6. 服务端将设备绑定到用户
7. 服务端记录设备注册审计事件

设备信息建议包含：

- 设备名称
- 操作系统
- 客户端版本
- 公钥指纹
- 首次注册时间

## 5. 服务列表数据流

1. 客户端携带访问令牌请求服务列表
2. 服务端解析用户和设备
3. 服务端检查用户和设备状态
4. 服务端读取角色授权
5. 服务端读取用户级允许/禁止覆盖
6. 服务端计算最终可访问服务
7. 服务端返回服务列表

服务列表中建议包含：

- 服务 ID
- 服务名称
- 服务分组
- 服务图标
- 服务状态
- 权限来源
- 最近访问时间

## 6. 服务访问数据流

1. 用户点击服务
2. 客户端请求服务访问入口
3. 服务端验证用户、设备、会话、服务、策略
4. 服务端返回受控访问 URL
5. 客户端用系统默认浏览器打开 URL
6. 浏览器请求网关入口
7. 网关再次验证会话和策略
8. 网关转发请求到内网服务
9. 网关记录访问审计事件

关键原则：

- 服务访问时不能只依赖“客户端已经展示了该服务”
- 网关必须在每次访问时再次校验
- 客户端不能传入任意上游地址

## 7. 后台配置变更数据流

以修改服务上游地址为例：

1. 管理员在后台编辑服务
2. 后台提交 PATCH 请求
3. 服务端验证管理员权限
4. 服务端校验上游地址格式
5. 服务端更新服务目录
6. 服务端记录审计事件
7. 后台刷新服务详情

失败分支：

- 权限不足：`POLICY_ACCESS_DENIED`
- 服务不存在：`SERVICE_NOT_FOUND`
- 上游地址无效：`SERVICE_UPSTREAM_INVALID`

## 8. 授权计算数据流

输入：

- 用户 ID
- 设备 ID
- 服务 ID
- 用户角色
- 用户级覆盖规则
- 服务状态

输出：

- 允许或拒绝
- 拒绝原因
- 权限来源

优先级：

1. 用户禁用，拒绝
2. 设备禁用，拒绝
3. 服务禁用，拒绝
4. 用户级禁止，拒绝
5. 用户级允许，允许
6. 角色授权，允许
7. 默认拒绝

## 9. 审计记录数据流

所有关键操作都进入统一审计入口。

审计写入时机：

- 登录成功或失败
- 设备注册或禁用
- 服务访问成功或拒绝
- 管理员配置变更
- 策略拒绝

审计记录必须与 `requestId` 关联。

## 10. 数据一致性要求

- 权限配置变更后，新访问应立即使用新策略
- 已签发令牌可通过权限版本号失效
- 设备禁用后，相关会话应被吊销
- 服务禁用后，所有用户访问都应拒绝

## 11. 失败处理原则

- 客户端展示用户可理解文案
- 服务端记录开发者可读错误
- 审计记录业务事件与错误码
- 不把上游服务不可用误判为权限不足
