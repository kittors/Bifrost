# 客户端架构与页面流转

## 1. 定位

桌面客户端是用户访问私有服务的小卡片式入口。它负责登录、设备信任、会话状态、服务入口展示和受控访问打开，但不负责接管系统网络，也不承担后台管理能力。

客户端必须同时支持：

- macOS
- Windows
- Linux

因此客户端架构应尽量使用跨平台稳定能力，避免依赖某个平台专有交互。

## 2. 技术形态

客户端采用：

- `Electron`
- `TypeScript`
- `React`
- `Tailwind CSS v4`
- `shadcn/ui`
- `electron-vite`
- `pnpm workspace`

客户端由三部分组成：

- Main Process
- Preload
- Renderer

## 3. Main Process 职责

Main Process 负责系统级能力边界内的事情：

- 创建应用窗口
- 管理应用生命周期
- 管理安全存储访问
- 管理系统托盘
- 处理外部链接打开
- 与 Preload 建立受控 IPC
- 处理自动更新入口，第一阶段可先预留

Main Process 不应承担：

- 业务页面渲染
- 直接调用所有业务 API
- 任意网络代理
- 系统代理设置
- DNS 或路由修改

## 4. Preload 职责

Preload 是 Main Process 与 Renderer 的安全桥。

允许暴露：

- 获取应用版本
- 读取安全存储中的必要状态
- 发起设备密钥相关受控操作
- 打开外部浏览器链接
- 订阅应用级事件

禁止暴露：

- 任意文件系统访问
- 任意 shell 执行能力
- 任意网络代理能力
- 未经过白名单的 IPC 通道

## 5. Renderer 职责

Renderer 是 React 应用。

负责：

- 登录页
- 设备状态页
- 服务列表
- 会话状态
- 设置页
- 主题切换
- 错误提示

Renderer 不直接接触系统安全存储，只通过 Preload 暴露的受控 API 获取必要信息。

## 6. 页面信息架构

第一阶段客户端应尽量保持简单，不做复杂页面体系。建议信息架构如下：

```text
desktop/
  login
  services
  account
  settings
  diagnostics
```

其中 `services` 是默认主视图，`account`、`settings`、`diagnostics` 应以紧凑面板、Tabs 或更多菜单承载，不做后台式侧边栏页面。

## 7. 页面说明

## 7.1 登录页

目标：

- 输入服务端地址
- 输入账号密码
- 登录
- 触发设备注册或设备挑战

主要状态：

- 未配置服务端地址
- 正在登录
- 账号密码错误
- 设备未注册
- 设备被禁用
- 登录成功

## 7.2 设备设置页

目标：

- 展示当前设备名称
- 展示设备绑定状态
- 展示设备指纹摘要
- 引导首次注册设备

设备状态：

- `not_registered`
- `pending`
- `trusted`
- `disabled`

## 7.3 服务列表页

目标：

- 展示当前用户可访问服务
- 支持分组与搜索
- 点击打开服务入口

服务项建议展示：

- 服务名称
- 服务分组
- 服务状态
- 最近访问时间
- 权限来源，角色授权或用户级覆盖

服务列表应采用单列紧凑列表，而不是大卡片网格。

## 7.4 服务详情页

目标：

- 展示服务说明
- 展示访问入口
- 展示当前用户访问状态
- 展示最近访问记录摘要

## 7.5 账号页

目标：

- 展示用户信息
- 展示角色
- 展示当前设备
- 提供退出登录

## 7.6 设置页

第一阶段设置项应保持克制：

- 主题模式
- 启动行为
- 诊断日志级别
- 服务端地址查看

不提供系统代理、路由、DNS 等网络设置项。

## 7.7 诊断页

目标：

- 展示服务端连接状态
- 展示当前会话状态
- 展示设备状态
- 展示最近错误 requestId

该页面用于排障，但不暴露敏感令牌。

## 8. 客户端主流程

## 8.1 首次启动

1. 用户打开客户端
2. 客户端检查是否已有服务端地址
3. 客户端检查本地是否已有设备密钥
4. 若没有设备密钥，准备注册流程
5. 用户登录
6. 服务端完成账号认证
7. 客户端完成设备注册或设备挑战
8. 客户端进入服务列表页

## 8.2 日常启动

1. 客户端启动
2. 读取本地安全存储中的刷新令牌和设备信息
3. 尝试刷新会话
4. 若成功，进入服务列表
5. 若失败，进入登录页

## 8.3 打开私有服务

1. 用户点击服务
2. 客户端请求服务访问入口
3. 服务端返回受控 URL
4. 客户端调用系统默认浏览器打开该 URL
5. 网关完成后续访问鉴权与反向代理

## 9. 本地网络约束

客户端第一阶段明确不做：

- 系统代理
- 全局代理
- DNS 接管
- 路由修改
- TUN/TAP
- 本地透明代理

如果未来增加本地回环代理，也必须默认关闭，并限定为项目专用端口与项目专用规则。

## 10. IPC 安全约束

Renderer 与 Main 之间的 IPC 必须白名单化。

要求：

- 每个 IPC 通道有明确名称
- 每个 IPC 通道有明确入参与返回值
- 不允许 `send(channel, ...args)` 形式暴露任意 channel
- 不允许 Renderer 直接访问 Node.js API

## 11. 客户端状态管理

建议状态分类：

- Auth State：登录、令牌、当前用户
- Device State：设备 ID、设备状态、设备指纹
- Service State：服务列表、服务分组、最近访问
- UI State：主题、侧栏折叠、搜索关键词

敏感信息不得直接保存在普通前端状态持久化中。

## 12. 错误处理

客户端必须统一处理：

- `401`：清理会话并回到登录
- `403`：展示账号、设备或策略拒绝原因
- `502/504`：提示目标服务不可用
- 网络错误：提示无法连接服务端

错误提示中应保留 `requestId`，便于管理员排障。

## 13. 第一阶段验收标准

- 三端都能正常启动客户端
- 登录、刷新、退出流程清晰
- 设备注册与设备禁用状态可见
- 可访问服务列表正确
- 点击服务后通过浏览器打开受控网关入口
- 不修改系统代理、DNS、路由
