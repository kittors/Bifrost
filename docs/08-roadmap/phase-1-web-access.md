# 第一阶段实施路线图

## 1. 阶段目标

第一阶段目标是实现 Web 私有服务的受控访问闭环。

用户必须：

- 安装客户端
- 使用后台配置的账号登录
- 设备处于可信状态
- 拥有目标服务访问权限

才能访问指定私有 Web 服务。

## 2. 交付边界

第一阶段交付：

- Monorepo 基础结构
- 设计系统基础
- 统一 API 契约
- Go 网关服务端
- Web 管理后台
- Electron 桌面客户端
- PostgreSQL 数据模型
- Web 服务反向代理
- 审计日志

第一阶段不交付：

- 全局 VPN
- SSH/TCP 代理
- SSO/OIDC/LDAP
- 移动端
- 浏览器插件
- 复杂审批流

## 5. 验收场景

## 5.1 正常访问

1. 管理员创建用户
2. 管理员创建角色
3. 管理员配置 GitLab 服务
4. 管理员授权角色访问 GitLab
5. 用户客户端登录
6. 用户看到 GitLab
7. 用户点击打开 GitLab
8. 网关成功代理访问

## 5.2 未授权拒绝

1. 用户登录客户端
2. 用户没有 Jenkins 权限
3. 用户尝试访问 Jenkins
4. 网关返回 `POLICY_ACCESS_DENIED`
5. 审计记录拒绝事件

## 5.3 设备禁用

1. 管理员禁用设备
2. 用户客户端刷新失败
3. 用户访问服务被拒绝
4. 审计记录设备拒绝事件

## 5.4 服务禁用

1. 管理员禁用 GitLab 服务
2. 已授权用户尝试访问 GitLab
3. 网关返回 `SERVICE_DISABLED`
4. 审计记录拒绝事件

## 6. 风险与控制

风险：

- GitLab 这类服务可能有路径、Cookie、WebSocket 特殊行为

控制：

- 先选择一个内部测试服务验证代理
- 再接 GitLab
- 单独验证大文件上传和 WebSocket
- WebSocket 需同时覆盖 Gateway 代理和 Desktop 本地回环代理，避免只验证服务端路径

风险：

- 客户端三端行为差异

控制：

- 不做复杂自绘窗口
- 不做系统代理
- 先保证登录、服务列表、打开浏览器稳定

## 7. 完成定义

第一阶段完成定义：

- 可通过客户端登录并访问被授权 GitLab
- 未授权访问被拒绝
- 设备禁用立即生效
- 服务禁用立即生效
- 后台可完成全部基础配置
- 审计可查关键事件
- 客户端未修改系统代理、DNS 或路由
