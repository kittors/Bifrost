# 安全模型与信任边界

## 1. 设计原则

Bifrost 的安全模型围绕以下原则构建：

- 默认拒绝
- 显式授权
- 设备可信
- 会话可吊销
- 访问可审计
- 本机最小侵入

这意味着系统不会把“知道一个入口地址”视为允许访问，也不会把“已经登录过一次”视为永久可信。

## 2. 信任边界

## 2.1 不可信边界

以下对象默认视为不可信：

- 公网网络环境
- 用户浏览器请求来源
- 外网设备的初始状态
- 任意未注册设备
- 任意未显式配置的目标上游地址

## 2.2 可信边界

以下对象可以在校验后进入受控可信范围：

- 已登录且未禁用的账号
- 已绑定且未禁用的设备
- 已签发且未过期、未吊销的会话
- 已在服务目录中配置的上游服务
- 已命中授权规则的访问请求

## 3. 加密与密码学模型

## 3.1 基本原则

本项目不采用“全部数据直接使用非对称加密承载”的设计。工程上应采用：

- 非对称密码体系用于设备身份、签名校验、密钥交换
- 对称加密或 TLS 用于实际业务传输

这样既安全，又具备可接受的性能。

## 3.2 设备密钥

客户端在首次注册或首次登录时生成设备密钥对。

建议：

- 签名算法采用 `Ed25519`
- 私钥保存在系统安全存储中
- 公钥上传至服务端与设备记录绑定

私钥不得以明文形式保存在应用可随意读取的普通配置文件中。

## 3.3 传输安全

所有客户端到网关、后台到服务端的交互都必须基于 `HTTPS/TLS`。

生产要求：

- 证书必须来自可信 CA 或企业可信根
- 禁止明文登录和明文后台 API
- 禁止通过 HTTP 传输令牌

## 4. 认证模型

## 4.1 账号认证

第一阶段使用后台配置的本地账号体系。

要求：

- 密码不得明文存储
- 使用现代密码散列算法，例如 `Argon2id`
- 支持账号启用、禁用、重置密码、强制下线

## 4.2 设备认证

设备是独立的安全主体。登录不仅要验证账号，也要验证设备状态。

推荐流程：

1. 客户端提交账号凭证
2. 客户端提供设备 ID 与设备公钥标识
3. 服务端发起挑战
4. 客户端使用设备私钥签名挑战
5. 服务端验签后确认设备身份

## 4.3 会话模型

建议采用：

- 短期访问令牌
- 较长期刷新令牌
- 刷新令牌必须绑定设备

令牌至少应绑定以下信息：

- 用户 ID
- 设备 ID
- 会话 ID
- 过期时间
- 权限版本号

当用户、角色、设备、服务状态发生关键变化时，应通过会话吊销或权限版本提升让旧令牌失效。

## 5. 授权模型

授权判断由以下维度共同构成：

- 用户状态
- 设备状态
- 服务状态
- 角色授权
- 用户级允许覆盖
- 用户级禁止覆盖

推荐优先级：

1. 若用户被禁用，拒绝
2. 若设备被禁用，拒绝
3. 若服务被禁用，拒绝
4. 若用户命中禁止覆盖，拒绝
5. 若用户命中允许覆盖，允许
6. 若角色允许，允许
7. 其他情况全部拒绝

## 6. 网关安全边界

网关不是任意代理，必须满足：

- 客户端不能指定任意内网地址进行转发
- 所有可访问目标必须来自后台服务目录配置
- 上游地址必须由服务端内部解析
- 请求路径只能映射到已配置服务上下文

目标是避免网关被滥用为“任意内网访问器”。

## 7. 审计模型

以下事件必须审计：

- 登录成功
- 登录失败
- 刷新令牌
- 设备注册
- 设备禁用
- 用户禁用
- 角色变更
- 服务新增、修改、启停
- 服务访问
- 策略拒绝
- 管理员操作

每条审计日志至少包含：

- 事件类型
- 事件时间
- 请求 ID
- 用户 ID
- 设备 ID
- 目标服务 ID
- 操作者
- 结果
- 拒绝原因或错误码

## 8. 客户端本机边界

为了确保不干扰用户本机网络环境，客户端必须遵守以下边界：

- 不修改系统代理
- 不修改系统 DNS
- 不修改系统路由
- 不创建 TUN/TAP
- 不注入浏览器代理配置
- 不静默劫持本地浏览流量

这条边界是产品级硬约束，不应因“实现方便”而破坏。

## 9. 常见攻击面与应对

## 9.1 凭证泄漏

应对：

- 密码安全散列
- 访问令牌短时有效
- 刷新令牌绑定设备
- 支持强制吊销会话

## 9.2 设备被复制或伪造

应对：

- 设备私钥存于系统安全存储
- 设备签名挑战验证
- 支持单设备禁用
- 审计设备指纹变化

## 9.3 网关被滥用访问任意内网

应对：

- 目标地址不由客户端输入
- 所有服务入口必须来自已配置服务目录
- 上游目标强校验

## 9.4 旧权限持续有效

应对：

- 短期令牌
- 权限版本号
- 显式会话吊销

## 10. 安全红线

- 不允许为了“更方便”而开放任意目标地址代理
- 不允许客户端在未登录时缓存可直接访问的私有服务入口
- 不允许后台与网关 API 存在未鉴权管理接口
- 不允许用本地明文文件保存私钥或敏感会话材料
- 不允许把服务可达性等同于授权成功
