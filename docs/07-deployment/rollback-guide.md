# 回滚方案

## 回滚原则

- 先保数据，再回应用。
- 不清楚 migration 影响时，不直接执行 destructive reset。
- 回滚过程中保持 requestId、错误日志和审计日志可追踪。

## Gateway 回滚

1. 保留当前故障版本镜像 tag。
2. 将部署配置切回上一个 Gateway 镜像。
3. 重启 Gateway。
4. 验证 `/healthz` 和 `/readyz`。
5. 验证管理员登录和客户端访问一个已授权服务。

## Admin Web 回滚

1. 将 Admin Web 镜像切回上一个版本。
2. 验证 `/health`。
3. 打开后台首页、用户页、服务页和审计页。
4. 确认 API 调用仍指向当前 Gateway。

## Desktop 回滚

1. 在内部发布渠道保留上一版安装包。
2. 通知试用用户卸载当前版本并安装上一版。
3. 若设备身份仍可用，用户重新登录即可恢复访问。
4. 如设备状态异常，管理员在后台重新启用或重建设备。

## 数据库回滚

优先顺序：

1. 若 migration 未执行，直接回滚应用镜像。
2. 若 migration 已执行但未写入生产流量，评估 down migration。
3. 若已写入生产流量，优先恢复发布前备份到新库并切换连接。

恢复后必须验证：

- 用户登录
- 角色授权
- 服务目录
- 审计查询
- 设备状态
